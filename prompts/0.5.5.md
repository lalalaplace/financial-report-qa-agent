# V0.5.5 总目标

新增：

```
intent_type = "rank_position_query"
```

支持用户问：

```
华润三九 2024 年营业收入排名第几？
片仔癀 2024 年净利润排第几？
云南白药 2024 年总资产从高到低排第几？
华润三九 2024 年资产负债率从低到高排第几？
片仔癀 2024 年净利率排名第几？
```

核心语义：

```
指定公司 + 指定年份 + 指定指标 + 排序方向 → 返回该公司在全体公司中的名次。
```

------

# 1. V0.5.5 支持范围

## 支持

```
1. 单公司
2. 单年份
3. 单指标
4. base 指标名次
5. derived 指标名次
6. 从高到低排名
7. 从低到高排名
8. 返回公司名次、指标值、参与排名公司数
```

## 不支持

```
1. 多家公司同时查名次
2. 多指标同时查名次
3. 同比增速排名第几
4. 区间增长率排名第几
5. 近三年增长排名第几
6. 指定公司集合内排名
7. 多年排名变化
8. 排名趋势
```

后续如果需要，可以拆成：

```
yoy_rank_position_query
trend_rank_position_query
rank_position_trend_query
```

但 V0.5.5 不做。3. intent 边界

需要在 `query_planner.md` 里写清楚边界。

| 用户问题                                      | intent                  |
| --------------------------------------------- | ----------------------- |
| 华润三九 2024 年营业收入是多少？              | point_query             |
| 华润三九 2024 年营业收入同比是多少？          | yoy_query               |
| 华润三九近三年营业收入趋势？                  | trend_query             |
| 华润三九和片仔癀 2024 年营收谁更高？          | company_compare_query   |
| 2024 年营业收入最高的前 10 家公司             | ranking_query           |
| 2024 年营业收入同比增速最高的前 10 家公司     | yoy_ranking_query       |
| 2022 到 2024 年营业收入增长最快的前 10 家公司 | trend_ranking_query     |
| **华润三九 2024 年营业收入排名第几？**        | **rank_position_query** |

核心判断：

```
如果问题要求“某家公司排第几”，走 rank_position_query。
如果问题要求“前 N / 后 N 公司是谁”，走 ranking_query / yoy_ranking_query / trend_ranking_query。
```

## 5.2 排序方向规则

| 用户表达         | rank_direction |
| ---------------- | -------------- |
| 排名第几         | desc           |
| 排第几           | desc           |
| 从高到低排第几   | desc           |
| 按高到低排第几   | desc           |
| 从低到高排第几   | asc            |
| 按低到高排第几   | asc            |
| 最低口径排名第几 | asc            |
| 越低越好排第几   | asc            |

默认：

```
未说明方向时，rank_direction = desc。
```

原因：大多数财务指标排名默认按数值从高到低。

但用户明确说“从低到高”时，必须用 `asc`。

# 7. slot validator 设计

新增：

```
agent/nodes/slot_validators/rank_position_validator.py
```

## 校验规则

```
1. intent_type 必须是 rank_position_query
2. companies 必须有且只有 1 个
3. metrics 必须有且只有 1 个
4. report_year 必须存在
5. time_mode 必须是 single_year
6. rank_direction 必须是 asc / desc
7. 不需要 limit
8. base / derived 指标都允许
```

## 错误类型建议

```
missing_company
multiple_companies_not_supported
missing_metric
multiple_metrics_not_supported
missing_year
unsupported_rank_position_time_mode
missing_rank_direction
unsupported_rank_position_metric_type
```

## validator 伪逻辑

```
def validate_rank_position_slots(state):
    plan = state.get("query_plan") or {}
    companies = state.get("companies") or []
    metrics = state.get("metrics") or []

    if len(companies) == 0:
        return fail("missing_company")

    if len(companies) > 1:
        return fail("multiple_companies_not_supported")

    if len(metrics) == 0:
        return fail("missing_metric")

    if len(metrics) > 1:
        return fail("multiple_metrics_not_supported")

    if not plan.get("report_year"):
        return fail("missing_year")

    if plan.get("time_mode") != "single_year":
        return fail("unsupported_rank_position_time_mode")

    if plan.get("rank_direction") not in ("asc", "desc"):
        return fail("missing_rank_direction")

    return ok()
```

## 8.1 base 指标名次 SQL

用窗口函数。

推荐用 `RANK()`，不是 `ROW_NUMBER()`。

原因：

```
如果两个公司指标值相同，它们应该拥有相同名次。
```

SQL 模板：

```
WITH ranked AS (
    SELECT
        c.company_code,
        c.company_name,
        t.report_year,
        t.report_period,
        t.<metric_field> AS metric_value,
        RANK() OVER (
            ORDER BY t.<metric_field> DESC
        ) AS rank_no,
        COUNT(*) OVER () AS total_count
    FROM <metric_table> t
    JOIN company_dim c
        ON t.company_id = c.company_id
    WHERE t.report_year = ?
      AND t.report_period = ?
      AND t.<metric_field> IS NOT NULL
)
SELECT
    company_code,
    company_name,
    report_year,
    report_period,
    metric_value,
    rank_no,
    total_count
FROM ranked
WHERE company_code = ?
```

如果 `rank_direction = asc`：

```
RANK() OVER (
    ORDER BY t.<metric_field> ASC
) AS rank_no
```

建议加稳定排序吗？

窗口函数里如果用 `RANK()`，相同值并列时不能再加 `company_code ASC`，否则并列会被打散。

所以：

```
RANK() OVER (ORDER BY metric_value DESC)
```

外层展示时可以加：

```
ORDER BY company_code ASC
```

但因为最后只查一个公司，外层排序没有必要。

------

## 8.2 derived 指标名次 SQL

复用 V0.5.1 derived ranking 口径。

同表 numerator / denominator 示例：

```
WITH metric_values AS (
    SELECT
        c.company_code,
        c.company_name,
        t.report_year,
        t.report_period,
        CAST(t.<numerator_field> AS DOUBLE)
            / NULLIF(CAST(t.<denominator_field> AS DOUBLE), 0) AS metric_value
    FROM <table> t
    JOIN company_dim c
        ON t.company_id = c.company_id
    WHERE t.report_year = ?
      AND t.report_period = ?
      AND t.<numerator_field> IS NOT NULL
      AND t.<denominator_field> IS NOT NULL
      AND t.<denominator_field> != 0
),
ranked AS (
    SELECT
        company_code,
        company_name,
        report_year,
        report_period,
        metric_value,
        RANK() OVER (
            ORDER BY metric_value DESC
        ) AS rank_no,
        COUNT(*) OVER () AS total_count
    FROM metric_values
)
SELECT
    company_code,
    company_name,
    report_year,
    report_period,
    metric_value,
    rank_no,
    total_count
FROM ranked
WHERE company_code = ?
```

跨表 numerator / denominator 时，使用 V0.5.1 已有 join 逻辑即可。

# 9. SQL Guard 检查

需要确认 SQL Guard 允许：

```
WITH CTE
RANK() OVER
COUNT(*) OVER
CAST
NULLIF
WHERE company_code = ?
```

仍然要限制：

```
SELECT only
白名单表
必须有 report_year 条件
必须有 report_period 条件
禁止 DDL / DML
禁止多语句
禁止无约束全表输出
```

rank_position_query 不一定需要 `LIMIT`，因为最终有：

```
WHERE company_code = ?
```

但如果 SQL Guard 强制所有排名类 SQL 必须有 LIMIT，需要为 `rank_position_query` 做受控例外：

```
rank_position_query 可无 LIMIT，但必须有：
1. report_year 条件
2. report_period 条件
3. company_code 过滤
4. 窗口排名表达式
```

不要为所有 SQL 放开 LIMIT 限制。
